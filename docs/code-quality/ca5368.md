# <a name="ca5368-set-viewstateuserkey-for-classes-derived-from-page"></a>CA5368：針對衍生自頁面的類別設定 ViewStateUserKey

|||
|-|-|
|CheckId|CA5368|
|類別|Microsoft.Security|
|重大變更|不中斷|

## <a name="cause"></a>原因

<xref:System.Web.UI.Page.ViewStateUserKey?displayProperty=nameWithType>未在或方法中指派屬性 <xref:System.Web.UI.Page.OnInit%2A?displayProperty=nameWithType> `Page_Init` 。

## <a name="rule-description"></a>規則描述

設計 ASP.NET Web Form 時，請注意跨網站偽造要求（CSRF）攻擊。 CSRF 攻擊可以將來自已驗證使用者的惡意要求傳送至您的 ASP.NET Web Form。

保護 ASP.NET Web Form 中的 CSRF 攻擊的其中一種方式，是將頁面的設定 <xref:System.Web.UI.Page.ViewStateUserKey> 為無法預測且對會話而言是唯一的字串。 如需詳細資訊，請參閱[利用 ASP.NET 的內建功能來至今尚未有人網路攻擊](/previous-versions/dotnet/articles/ms972969(v=msdn.10)#viewstateuserkey)。

## <a name="how-to-fix-violations"></a>如何修正違規

<xref:System.Web.UI.Page.ViewStateUserKey>針對每個會話，將屬性設定為無法預測和唯一的字串。 例如，如果您使用 ASP.NET 會話狀態， <xref:System.Web.SessionState.HttpSessionState.SessionID?displayProperty=nameWithType> 將會生效。

## <a name="when-to-suppress-warnings"></a>隱藏警告的時機

在下列情況下，您可以放心地隱藏此規則中的警告：
- ASP.NET Web Form 頁面不會執行敏感性作業。
- 跨網站偽造要求攻擊會以此規則不會偵測到的方式降低。 例如，如果頁面繼承自包含 CSRF 防禦的主版頁面。

## <a name="pseudo-code-examples"></a>虛擬程式碼範例

### <a name="violation"></a>違規

```csharp
using System;
using System.Web.UI;

class ExampleClass : Page
{
    protected override void OnInit (EventArgs e)
    {
    }
}
```

### <a name="solution"></a>解決方法

```csharp
using System;
using System.Web.UI;

class ExampleClass : Page
{
    protected override void OnInit (EventArgs e)
    {
        // Assuming that your page makes use of ASP.NET session state and the SessionID is stable.
        ViewStateUserKey = Session.SessionID;
    }
}
```
