## <a name="data-visualization"></a><a name="BKMK_DataVisualization"></a>資料視覺效果

### <a name="overview"></a>總覽
 圖表是一種視覺化的方式，可匯總資料並加以視覺化，以加強決策制定。 他們可以協助使用者面臨大量的資料，但這一點很重要，那就是值得注意的是什麼，以及可能需要採取行動的事項。

 如果下列任一條件成立，則使用者將受益于圖表：

- 圖表會協助使用者識別他們可以採取行動的工作嗎？

- 圖表是否可讓使用者預測潛在變更的結果？

- 圖表會協助使用者探索趨勢並識別模式嗎？

- 圖表是否可讓使用者做出更好的決策？

- 圖表會協助回答使用者可能在指定內容中有的特定問題嗎？

#### <a name="general-rules-for-charts"></a>圖表的一般規則

- 清楚標示資料。 沒有說明的圖例只是不錯的圖片。

- 從零開始軸，以避免扭曲比例。 「行長度」和「橫條圖大小」是瞭解資料點之間關聯性的重要視覺提示。

- 建立圖表，而不是資訊圖。 資訊圖是資料的藝術標記法，其主要目標是視覺故事行銷。 圖表可以 (，而且應該) 以視覺化的方式呈現，但讓資料自己說話。

- 避免 skeumorphism、圖形橫條圖、對比井號和其他資訊圖觸控。

- 請勿使用3D 效果做為裝飾元素。 只有當他們真的對使用者理解資訊的能力而言，才使用它們。

- 避免使用多行和填滿，因為兩個以上的色彩可能會使這類圖表很容易閱讀和解讀。

- 請勿使用圖表 (或任何圖例) 做為瞭解概念或與資料互動的唯一方法。 這對視力受損的使用者造成了困難。

- 請勿使用圖表做為頁面上的不必要或裝飾元素。 換句話說，如果圖表並未加入任何值或協助使用者解決問題，請不要使用它。

### <a name="chart-types"></a>圖表類型
 在 Visual Studio 中使用的圖表類型包括橫條圖、折線圖、修改過的圓形圖（稱為環形圖或「環圈圖」）、時間軸、散佈圖 (也稱為「群集圖表」 ) 和甘特圖。 每種類型的圖表都適用于通訊不同類型的資訊。

### <a name="other-charting-considerations"></a>其他圖表考慮

#### <a name="color"></a>Color

## <a name="on-object-ui-and-peeking"></a><a name="BKMK_OnObjectUI"></a>物件內部 UI 和查看
 本節提供要查看的內容，也稱為程式碼查看視圖，這是一種 Visual Studio 唯一的特定物件 UI 類型。

### <a name="overview"></a>總覽

- 物件上的 UI 應該提供使用者更多的資訊或互動性，而不需要從其主要工作注意力。

- Visual Studio 中的內建物件 UI 主要模式就是所謂的「資訊在關注點」。

- Visual Studio 中的內建物件 UI 是內嵌或浮動的，也可能是永久性或暫時性的。

  - 程式碼查看視圖是 Visual Studio 中的一種物件型 UI，是內嵌和持久的。

  - CodeLens，Visual Studio 中的物件型 UI 類型為浮動和暫時性

  瞭解某段程式碼的運作方式，或尋找該程式碼的詳細資料時，通常需要開發人員切換內容並移至其他內容或另一個視窗。 這些內容轉移可能會造成干擾，因為如果使用者離開其主要視窗，可能會失去其原始工作的焦點。 此外，取得原始內容可能很麻煩，特別是當切換 windows 造成其原始程式碼被其他 UI 遮蔽時。

  物件上的 UI 會遵循名為「資訊的觀點」模式。 這些訊息、快顯視窗和對話方塊提供使用者其他相關資訊，可新增清楚或互動性，而不會失去其主要工作的焦點。 物件上的 UI 範例包括當使用者將指標停留在通知區域中的圖示上時所顯示的快顯視窗、拼錯字之下的紅色波浪線，以及 Visual Studio 2013 中引進的查看視圖。

### <a name="decision-points"></a>決策點
 在 Visual Studio 中，有幾種方式可以在注意時使用這種模式的資訊。 選擇正確的機制，並以一致且可預測的方式來實行，對整體體驗而言是不可或缺的。 否則，使用者可能會有令人困惑或不一致的體驗，而無法從內容本身會使焦點。

#### <a name="relationships-between-master-and-detail-content"></a>主要和詳細內容之間的關聯性
 重點處的資訊是用來顯示內容之間的關聯性，使用者將焦點放在 (「主要」內容) 以及其他相關內容 (「詳細資料」內容) 。 在此模式中，詳細內容會清楚地與使用者正在使用的內容相關，而且可以顯示在主要內容的附近。 不需要擔心主要內容的補充資訊或資訊，應遵循另一個模式，例如工具視窗。

- **一律**以靠近主要內容的鄰近性顯示詳細內容。

- 請**務必**確定詳細內容仍然讓使用者保持專注于主要內容。 通常，達到此目標的最佳方式，就是盡可能地將詳細內容轉譯為接近主要內容。 這可以藉由在主要內容旁的快顯視窗中轉譯詳細內容來完成，或藉由在主要內容底下轉譯詳細內容來呈現。

- **永遠不要**在將使用者帶離主要內容的重點處使用資訊。 如果使用者需要個別查看詳細資料內容，請公開可讓使用者執行此動作的明確動作。

#### <a name="design-details"></a>設計詳細資料
 一旦判斷出物件上的 UI 是正確的選擇之後，有四個主要的設計考慮：

1. **持續性：** 內容是否預期為持久或暫時性？
   使用者是否想要讓資訊保持可見或與其互動？ 或者，使用者是否想要快速概覽資訊，然後繼續進行其主要工作？

2. **內容類型：** 內容是資訊、可採取動作還是導覽？
   使用者需要有關主要內容的其他詳細資料嗎？ 使用者是否需要完成影響主要內容的工作？ 還是需要將使用者導向至另一個資源？

3. **指標類型：** 環境指標是否合理？
   這項資訊是以實用的方式摘要，而且在沒有主要內容的情況下顯示？

4. **筆勢：** 會使用哪些手勢來叫用和關閉 UI？
   使用者會如何顯示詳細內容，並將其傳送出去？ 加入手勢（例如釘選以在暫時性和永久性狀態之間切換）是否有任何價值？

   這四個決策點都會影響內建物件 UI 的主要元件。

### <a name="on-object-ui-components"></a>物件上的 UI 元件

1. 容器 (內容呈現者) 類型

    - 浮動

    - 內嵌

2. 內容類型

    - 資訊：可能是靜態或動態的資料

    - 可採取動作：變更主要內容的命令

    - 導覽：將使用者帶到另一個視窗或應用程式的連結，例如 MSDN

3. 軌跡

    - 引動過程

    - 關閉

    - 釘選

    - 其他互動

4. 持續性和認可模型

    - 暫時性

    - 持久

    - 自動

    - 隨選

5. 環境指示器 (選擇性) 

    - 波浪線波浪線

    - 智慧標籤圖示

    - 其他環境指標

#### <a name="container-content-presenter-type"></a>容器 (內容呈現者) 類型
 有兩個主要選項可用來在注意時呈現內容：

1. **內嵌：** 內嵌呈現者，例如在 Visual Studio 2013 程式碼編輯器中引進的 [查看] 視圖，會藉由移動現有內容來為新內容提供空間。

    - 如果您預期使用者會想要花相當長的時間來參考或與您所呈現的內容互動，則**偏好使用**內嵌展示者。

    - 如果您預期使用者會想要查看您所呈現的資訊，請**避免**內嵌展示者，然後在最少中斷的情況下繼續執行主要工作。

2. **浮動：** 浮動展示者會盡可能靠近選取的內容，但不會改變現有內容的版面配置。 可以運用各種策略，例如將浮動內容面板顯示在最接近的可用空白字元上，以選取的符號。

    - 如果您預期使用者會想要查看您所呈現的資訊，請**偏好**使用浮動的展示者，然後在最少中斷的情況下繼續執行主要工作。

    - 如果您預期使用者會想要花相當長的時間來參考或與您所呈現的內容互動，請**避免**浮動展示者。

#### <a name="content-type"></a>內容類型
 有三種主要類型的內容可以在任何物件的 UI 容器內顯示。 可以顯示這些資訊類型的任何組合。 這三種類型為：

1. **資訊：** 大部分的內建物件 UI 容器都會顯示一些資訊內容。 內容可以代表環境目前狀態的相關資訊，也可能代表環境未來狀態的相關資訊。 例如，它可以用來在現有的程式碼上顯示特定命令的效果，例如重構。

    - 請**一律**使用您所顯示之資訊的標準標記法。 例如，程式碼看起來應該像程式碼，以語法醒目提示完成，而且應該遵守使用者所設定的任何字型和其他環境設定。

    - 如果該相同資訊呈現為主要內容，請**務必**針對資訊性內容，考慮支援任何可能的動作。 例如，如果在物件 UI 容器內呈現現有程式碼，則強烈考慮支援流覽和修改該程式碼的功能。

    - 如果呈現的資訊內容代表未來可能的狀態，請**務必**考慮使用不同的背景色彩。

2. 可操作：某些物件上的 UI 容器將提供對主要內容執行一些動作的功能，例如執行重構作業。

    - **一律**將可操作的命令與資訊性內容分開放置。

    - 適當時**一律**啟用和停用動作。

    - 請**一律**參考標準指導方針，以代表對話方塊內的命令。

    - 請**一律**將在物件 UI 容器中公開的動作數目保持為絕對最小值。 與內建物件 UI 互動應該是輕量的快速體驗。 使用者不應該在管理物件上的 UI 容器本身時耗費能源。

    - 請**務必**考慮如何及何時關閉或關閉物件上的 UI 容器。 作為最佳作法，當叫用該動作時，主要和詳細內容之間的對話結束的任何動作也應該關閉物件上的 UI 容器。

3. **導覽：** 某些物件上的 UI 容器包含將使用者帶到另一個視窗或應用程式的連結，例如在使用者的網頁瀏覽器中開啟 MSDN 文章。

    - **一律**在任何導覽連結前面加上 "Open"，讓使用者不會因為流覽至其他內容而感到驚訝。

    - **一律**將導覽連結從可操作的連結中分開。

#### <a name="ambient-indicators-optional"></a>環境指示器 (選擇性) 
 環境指標可能很微妙，包括從其他程式碼的對比色彩呈現的文字，或明顯包含 tickler 符號（例如波浪線底線和智慧標籤圖示）。 環境指標會傳達其他相關資訊的可用性。 在理想的情況下，即使不需要使用者與他們互動，也會提供有用的資訊。

- **一律**放置環境指標，使其不會干擾或塞滿使用者。 如果無法以這種方式定位環境指標，請考慮另一個解決方案。

- 請**一律**盡可能將環境指標放在靠近其相關內容的位置。

- 請**一律**嘗試建立一個指標，以摘要說明它所提供的資訊。 請考慮提供可用資料項目數目的計數 (例如「3個參考」，而不只是「參考」 ) 或考慮其他摘要資料的方式。

  - 如果指標的資料不一定會計算並顯示，則會立即考慮在計算值時提供漸進式的意見反應。 例如，請考慮以動畫顯示可反映可用資料更新的變更，類似于 Windows Phone 的電子郵件動態磚會隨著未閱讀的電子郵件數目增加而重新整理。

- **絕對不要**新增更多指標，而不是使用者可合理接受的特定內容片段。 環境指標應該很有用，而不需要使用者進行任何互動。 如果指標需要溢位和其他管理控制項使其變成可供觀看，則指標會失去其環境。

#### <a name="gestures"></a>軌跡
 允許使用者將焦點放在主要內容上的主要層面，就是支援適當的手勢來開啟和關閉其他詳細資料內容。

- **一律**要求使用者執行一些明確的手勢來開啟其他內容。 常見的開放手勢包括：

  - 暫留 **：** 工具提示或非互動式資訊內容

  - **Explicit 命令：** 內嵌展示者

  - **按兩下環境指標：** CodeLens 快顯視窗

- 每當使用者按下 Esc 鍵時，**一律**關閉詳細內容。

- 請**一律**考慮內建物件 UI 的內容。 對於允許在容器內進行互動的內容展示者，請仔細考慮是否要在停留時顯示其他資訊，這可能會干擾使用者的工作流程。

- **永遠不**會在暫留的內容上顯示可編輯或邀請使用者互動的內容。 如果使用者嘗試將游標移到詳細資料內容上，這種行為可能會讓使用者感到挫折，因為工具提示的標準行為是在游標不再超過產生它的主要內容時立即關閉。

## <a name="selection-models"></a><a name="BKMK_SelectionModels"></a>選取模型

### <a name="overview"></a>總覽
 選取模型是一種機制，用來指示和確認在使用者介面中感關注的一個或多個物件上的作業。 本主題討論 Visual Studio 檔編輯器中的選取範圍互動模式：文字編輯器、設計介面和模型介面。

 使用者必須有指出要 Visual Studio 其工作的方法，而且 Visual Studio 必須以可預測的方式回應使用者有關其運作內容的意見反應。 使用者與使用者介面之間的差異或 miscommunication 可能會導致使用者不會注意到動作，這可能會造成非預期的結果。 通常，除非使用者發現某個專案遺失或已變更，否則不會察覺到此錯誤。 因此，選取模型就是使用者介面設計的其中一個最重要的部分。 雖然 Visual Studio 中的選取模型與 Windows 一致，但還是有些許差異。

 在 Visual Studio 中，如同在 Windows 中，選取模型會根據發生互動的內容而有所不同。 選擇可能會發生在四種類型的物件中：

- Text

- 圖形物件

- 清單和樹狀結構

- 方格

  在這些物件中，有三種類型的選擇：

- 臨近

- 無

- 區域

#### <a name="scope"></a>影響範圍
 選取專案最重要的部分是確保使用者知道其運作所在的視窗 (啟用) 以及焦點位於 (選取) 的位置。 Visual Studio 擴充 Windows 中的視窗管理功能，但啟用配置是相同的：與視窗互動會將焦點帶入視窗。 Visual Studio 有兩個啟用指示器：一個用於文件視窗，另一個用於工具視窗。

 針對文件視窗，使用中視窗是由位於前端的文件視窗索引標籤所表示，並會變更其背景色彩：

 **作用中索引標籤選項**

### <a name="graphical-selection"></a>圖形選取

#### <a name="interaction"></a>互動
 繪圖物件選取可能很複雜，而且取決於數個因素：

- **編輯器的主要選取模型。** 包含繪圖物件的編輯器也可以用來編輯文字或格線。 例如，編輯器可能是以文字為基礎的編輯器，也支援繪圖物件的位置，例如 Visual Studio XAML 設計工具。 支援多個物件類型可能會影響使用者選取由不同物件類型組成之群組的方式。

- **支援主要和次要選取狀態。** 編輯器可以提供主要和次要選取狀態，讓物件可以一致地進行編輯、配合彼此調整、調整大小等等。

- **就地編輯支援。** 編輯器也可以允許編輯繪圖物件的內容。 例如，「矩形」圖形可能也會包含內的文字，可由使用者進行變更。 此外，該文字可以置中或對齊。 就地編輯牽涉到更詳細的使用者互動層級，因此需要一組適當的視覺提示來向使用者呈現狀態資訊。

#### <a name="selection-within-lists-and-grid-controls"></a>清單和方格控制項內的選取範圍
 清單選取專案中的主要概念是它是以資料列為基礎，這表示當選取專案時，會選取整個資料列做為一個單位。 相較之下，方格可讓您選取特定資料格，而不會影響資料列的任何其他層面。 方格也可能包含嵌套資料列的階層 (例如在 TreeGrid) 中，可讓您透過與父資料列互動來選取和取消選取階層的整個分支。 清單中的選取專案會以簡單的醒目提示色彩顯示在整個資料列上。 如果所有資料格都是) 唯讀的，則焦點會以單一圖元的圓角框線顯示在目前可編輯的資料列或資料格 (資料列上。

> [!NOTE]
> **焦點**和**選取範圍**是不同的概念。 *焦點*是指出哪個 UI 元素的目標是要接收未明確導向另一個物件的輸入，而*選取範圍*則是指物件在可能發生後續作業的一組物件中包含的狀態。

 清單中的選擇可以是連續、不相鄰或區域。 允許多重選取時，應該一律支援連續和不連續的選取，而 [區域 (] 方塊的支援) 選取專案是選擇性的。 藉由拖曳清單主體的空白字元來起始區域選取。

| Object | 選取 |
|--------|------------|
| 清單 | 臨近 |
| 清單 | 無 |
| 清單 | 區域 |

 按一下清單上的 [一次]，即可選取發生點擊的資料列。 如果使用者在支援就地編輯的清單儲存格中按一下，則資料格也會立即啟用以進行就地編輯。 否則，會立即選取整個資料列，並顯示反白顯示。

 在清單主體中拖曳會執行下列三項動作之一：

- 如果清單支援區域選取，而且滑鼠向下鍵在空白字元中，則起始

- 如果清單儲存格或資料列支援拖曳來源，則起始拖放作業

- 選取目前的資料列

##### <a name="in-place-editing"></a>就地編輯
 允許就地編輯時，有兩個基本模型：簡單的編輯控制項和屬性選擇器。 使用簡單的編輯控制項，只要啟用就地編輯，就會反白顯示內容並準備好進行使用者輸入。 當屬性選擇器執行時，叫用屬性選擇器的按鈕會在就地編輯模式啟動時顯示，而且不會反白顯示目前的選取範圍。 儲存格中的 [選擇器] 按鈕應該靠右對齊。 如需就地編輯範例，請參閱 Visual Studio 中的 [**屬性] 視窗**和 [**工作清單**]。

##### <a name="keyboard-support"></a>鍵盤支援
 清單和格線中選取的鍵盤支援遵循標準 Windows 慣例：

- 方向鍵：瀏覽清單，並在移動焦點時選取每個資料列/資料格。

- Shift + 箭號會以方向鍵的方向執行連續的選取。

- Ctrl + 箭號後面接著空格鍵會在從選取範圍內新增和移除清單專案之間切換，建立脫離的選取範圍。

- 針對包含嵌套階層的格線，向右鍵會展開父資料列，而向左鍵會折迭一個。

- 如果可以編輯資料格，Tab 鍵會將焦點移到目前資料列中的資料格之間。

- Enter 鍵會在清單中的專案上執行預設命令， (通常會**開啟**) 。

- F2 鍵會針對目前選取的儲存格，啟用就地編輯。

### <a name="window-configurations"></a>視窗設定
 視窗設定是開發環境的基本呈現方式，它是一種配置，其中包含目前的工具視窗清單，以及它們的相片順序。 針對 IDE 所管理的 windows (IDE windows) ，每位使用者會保存版面配置資訊，因此當使用者啟動 IDE 時，視窗配置會顯示為 [上次結束時] Visual Studio。 IDE 視窗的狀態和位置是以 XML 格式保存在自訂的選項檔案中。 載入 IDE 的封裝所建立的工具視窗會將其狀態資訊保存在登錄中，而且不一定是每位使用者。

#### <a name="profile-specific-layouts"></a>設定檔特有的版面配置

## <a name="touch-input"></a><a name="BKMK_TouchInput"></a>觸控輸入
 使用者越來越容易在觸控裝置上使用 Microsoft 開發產品。 不過，有一些阻礙讓您難以在觸控裝置上使用開發工具。 使用者會預期我們的產品提供可靠且精確的觸控體驗。 這些指導方針的目的是要通知有關要納入哪些觸控功能，以及鼓勵 Visual Studio 和相關產品之間一致觸控體驗的決策。

### <a name="levels-of-experience"></a>體驗層級
 下列經驗層級的目的是為了協助團隊根據其所需的觸控投資程度來決定要提供哪些觸控功能。

- **基本經驗**適用于想要提供觸控功能的小組，讓他們的工作不會有任何作用中的結束。

- **最佳體驗**適用于想要提供最常見觸控功能的小組 (例如，通常可在網際網路瀏覽器應用程式) 中使用。

- 更**高的經驗**適用于想要新增功能（例如手勢或其他選擇性功能）的小組，讓他們的應用程式更容易觸控。

||基本體驗|優化體驗|更高的體驗|
|-|----------------------|--------------------------|-------------------------|
|**可讓使用者 .。。**|修正程式碼和解決方案/專案層級的讀取，但不含無作用結束|執行維護、重構和流覽工作|自信地以一致、直覺且流暢的體驗操作|
|**編輯器**|觸控式移動和選取<br /><br /> 捲軸觸控以跳到並按 + 拖曳|縮小縮放比例<br /><br /> 快速滾動<br /><br /> 選取<br /><br /> 快顯功能表的輕鬆使用||
