# <a name="visual-studio-sdk-glossary"></a>視覺化工作室 SDK 字彙表
此術語表為[!INCLUDE[vsipsdk](../extensibility/includes/vsipsdk_md.md)]文件中使用的術語提供定義。

## <a name="terms"></a>詞彙
 外接程式 添加到主應用程式的實用程式應用程式、驅動程式或其他軟體。 在 Visual Studio 整合式開發環境 (IDE) 中,外接程式是基於自動化的應用程式,可擴展 IDE 的功能。

 自動化模型在 Visual Studio 的早期版本中稱為擴充性模型,是一個程式設計介面,允許您存取驅動 IDE 的基礎例程。 載入項、精靈和宏使用自動化模型中的物件來控制或擴展 IDE 的功能。

 命令 UI 上下文與 UI 命令或元素(如工具列)的可見性關聯。 命令 UI 上下文與選擇上下文不同,因為它不附加到視窗。

 命令 UI 內文可用於:

- 將 GUID 分配給啟動特定視窗時顯示的工具列。

- 將 GUID 分配給命令的可用性,而無需載入或運行 VSPackage。

- 分配 GUID 以影響活動密鑰綁定。

- 分配 GUID 以打開宏錄製。

- 分配 GUID 以啟動除錯模式或在編輯器中的設計和運行模式之間切換。

  元件軟體,可以成為應用程式功能的一部分,而該應用程式沒有任何關於軟體實現的任何預先存在的資訊。 元件和應用程式之間的通信僅透過 OLE 樣式介面進行。

  元件管理員 A`SOleComponentManager`服務 ,它為頂級元件提供非使用者介面協調服務。 服務`SOleComponentManager`服務`IOleComponentManager`服務服務 。

  元件 UI 管理`SOleComponentUIManager`員 A 服務,提供使用者介面協調服務。 服務`SOleComponentUIManager`實現和`IOleComponentUIManager``IOleInPlaceComponentUIManager`介面。

  上下文包`IVsUserContext`附加到環境元件的物件 (COM 物件)。 此物件包含尋找關鍵字 **、F1**關鍵字和與元件相關的屬性。 此外,上下文包還會指向連結到它們的任何子上下文包。

  上下文提供程式 IDE 中具有與其關聯的上下文包的元件。 此類元件包括工具視窗、編輯器或專案層次結構。

  設計器允許使用者操作 UI 的元素(表單、按鈕和其他控制件)的程式設計介面。

  DocData A COM 物件封裝文件的基礎數據,其中存在文檔/檢視分離(例如,在文字編輯器案例中,這將是所有文本編輯器視圖的基礎文本緩衝區)。 如果編輯器工廠不提供此物件,IDE 將代表它製造一個物件。 此物件的責任是管理資料持久性和共享語義的多個檢視在此相同`DocData`。 如果`DocData`物件`IOleCommandTarget`支援介面,則該介面包含在 UIShell 的命令路由中。

  文檔物件技術用於在主機提供的幀中託管 UI。 更具體地說,此術語是指支持`IOleDocument`和相關介面的任何嵌入。 該技術具有許多潛在的應用,如 COM 文件的實現詳細資訊、Visual Basic 5.0 中的工具視窗、Visual Basic 6.0 中的 ActiveX 設計器等。

  文件 用於將文件作為一個整體進行泛泛參考`DocData`,`DocView`和 。 例如,DocumentFrame 包含`DocView`, 但它`DocData`也保留對 處理持久性的引用。

  文件檢視使用者與之互動的文件物件/嵌入/視窗窗格以檢視並操作基礎`DocData`。 使用者不會利用`DocObject`作為界面設計一部分的文檔/視圖分離。 使用者使用整個 DocObject 充當檢視,而不是使用稱為的基礎資料更抽象(不太正式)的概`DocData`念 。 `DocView`對象始終嵌入 IDE 的文件框物件(MDI 子視窗)。

  `DTE` DTE(開發工具擴充性)對像是 Visual Studio 自動化模型中存取點最多的物件,它允許您以程式設計方式自動執行和擴展 IDE。

  由 IDE 實現的動態幫助視窗工具視窗,並顯示查找關鍵字或**F1**幫助主題的清單。

  以建立`DocView`的編輯器代碼(類別、CLSID) 如果支援視圖`DocData`和數據分離,它也會實現。

  擴展修改、自定義或添加到 IDE 的功能。 使用自動化模型或 VSPackages 創建擴展。

  外部編輯器 不是特定於 IDE 的編輯器,如 Microsoft Word。 它已通過自己的機制註冊,可在IDE之外使用。 如果可以嵌入此編輯器,它將顯示在 IDE 視窗中。 如果無法嵌入,則創建單獨的頂級視窗。

  層次結構節點樹,每個節點與一組屬性關聯。

  獨立的頂級元件一個元件,它使用無模式的頂級視窗,可以有效地作為獨立應用程式視窗運行,但作為進程內對象實現。 因此,獨立的頂級元件必須協調模式和消息迴圈服務與IDE。 進程內物件沒有自己的消息迴圈。

  資訊提供程式 資訊提供者是一個模組,可以查找關鍵字並返回主題清單,`IVsUserContextItem`以物件的形式。 要為資訊提供者提供**F1**和查找關鍵字項,請註冊已編譯的説明檔 *(。HxS*) 與系統。 這些檔案中的「說明」 主題提供動態幫助視窗中顯示的主題清單,並顯示使用者是否按**F1**。

  就地元件 VSPackage 物件,該`IOleInPlaceComponent`物件 實現介面來管理視覺上包含在 IDE 擁有的文檔視窗中的視窗。 就地元件不參與標準 OLE 功能表合併;相反,他們將其用戶介面元素集成到 IDE 中。

  有兩種類型的就地元件:硬接線就地元件和元件控制項。

  硬連線就地元件具有功能表、工具列和命令,它們緊密整合到 IDE 的使用者介面中,看起來好像它們直接內置到 IDE 中。

  元件控制者沒有將任何自己的使用者介面元素整合到 IDE 中;因此,元件控制項沒有將任何使用者介面元素整合到 IDE 中。相反,他們使用IDE的功能表、命令和工具列。 例如,「粗體」命令可用於在嵌入窗體的富文本控件中粗制所選單詞。 但是,元件控制項可以請求顯示動態安裝的特定於元件的 UI 元素。

  語言服務 一組物件,允許 VSPackage 開發人員實現計算機語言代碼編輯器的功能,如文本標記和著色。

  雜項檔專案專案用於容納不在任何專案中的打開檔。 此專案中的項目清單不會持久化。

  專案專案由層次結構物件或實現介面的`IVsHierarchy`COM 物件組成。

  專案特定的設計器或編輯器不能獨立於專案類型的設計器。 所有特定於項目的設計人員都必須在註冊表中輸入其編輯器工廠資訊。 然後,每當在特定專案中打開某個文件類型時,IDE 就可以實例化設計器。

  項目類型視窗 不斷追蹤當前活動專案層次結構和全域選擇上下文中的項的視窗。 項目類型視窗使用`SVsTrackSelectionEx`服務來提醒 IDE 更改並顯示向使用者的反饋。 解決方案資源管理員是項目類型視窗的範例。

  屬性視窗以前屬性瀏覽器。

  基於引用的項目專案,它不需要專案的檔位於同一目錄中。 相反,對來自其他不相關目錄的檔的引用由專案本身存儲和維護。

  運行文件表 內部結構,IDE 通過該結構維護所有當前打開的文檔的清單。 該清單包括記憶體中的所有打開的文檔,而不管當前是否正在編輯這些文檔。 文件是保存的任何項,包括在編輯器中打開的存儲過程、專案中的檔案或主專案檔(例如,*.vcproj 檔)。

  選擇上下文 數據,它是 IDE 中每個視窗的詳細資訊的一部分,用於追蹤活動選擇。 選擇內容包含:

- 指向專案層次結構`IVsHierarchy`介面的指標

- 項目項的項標識碼。

- 指向提供對`ISelectionContainer`活動物件屬性訪問的介面的指標。

- 元素值的陣列。

  服務 一組 COM 介面的協定,這些介面位於單個 COM 物件中。 創建由 GUID 識別的服務時,可以定義執行服務的 COM 介面集。 COM 物件使用服務相互通信。

  解決方案使用者使用的相關項目組。

  標準設計器一種設計器,可以獨立於專案類型使用。 所有標準設計人員必須在註冊表中輸入其編輯器工廠資訊。 然後,每當打開具有特定擴展名的檔時,IDE 就可以實例化設計器。 數據必須保存到檔中。

  標準編輯器,可獨立於任何特定專案類型使用。 此類編輯器在註冊表中註冊了編輯器工廠。 這允許 IDE 搜尋和調用編輯器。

  標準作業系統編輯器不是特定於可視化工作室的嵌入。 它是使用眾所周知的 Win32 密鑰註冊的(例如,Win32 資源管理器知道如何調用)。 如果可以嵌入這樣的編輯器,編輯器仍然顯示在 IDE 中的位置。 否則,將為此編輯器創建單獨的頂級視窗。

  子上下文包`IVsUserContext`鏈接到上下文包的物件。 物件包含 IDE 元件中所選內容的查找關鍵字 **、F1**關鍵字和屬性。 子上下文的範例包括工具視窗中的命令或編輯器中的關鍵字。

  工作清單 由 IDE 實現並顯示活動任務的清單的工具視窗。

  文字緩衝區`VSTextBuffer`物件的常見名稱 。

  文字檢視 物件`VSTextView`的常見名稱 。

  工具頂級元件一個元件,作為無模式彈出窗口運行,與IDE的使用者介面緊密協調。 與獨立的頂級元件一樣,工具頂級元件還必須將模式和消息迴圈服務與 IDE 協調。

  頂級元件 A VSPackage 物件,用於管理無模式的頂級視窗,而不是 IDE 視窗的工作區。 頂級元件實現`IOleComponent`介面,以利用消息循環服務,如訪問空閑時間。

  UI 活動 A VSPackage 物件可見且當前具有焦點。

  UI 層次結構`IVsUIHierarchy`實現 介面以允許顯示層次結構的 COM 物件。 UI 層次結構視窗`ISelectionContainer`實現 介面以更新屬性視窗;如果需要,其他項目類型視窗可以使用此實現。

  VSCT 可視化工作室命令表。 .vsct 檔包含有關功能表、工具列和指令以 XML 格式放置和行為的資訊。

  VSPackage 可安裝的軟體,通過貢獻以下一個或多個專案來擴展 Visual Studio IDE:使用者介面、服務、專案類型或編輯器/設計器。 VSPackage 由`IVsPackage`實現 介面的 COM 物件以及實現其他介面以支援選擇和其他功能的一個或多個其他 COM 物件組成。 此外,VSPackage 具有特定的註冊要求。
